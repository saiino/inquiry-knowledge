# 既知障害・一時情報(known issues)

**「今だけ正しい情報」の置き場。** FAQ(恒久ナレッジ)とは分けて管理する。
Microsoft側の障害・計画メンテ・社内側の一時的な設定変更など、同じ問い合わせが短時間に集中する事象をここに書く。

## 運用ルール

- inquiry-assistant は**FAQより先にこのファイルを確認**し、該当すれば「既知の事象」として即回答案を出す(個別調査をしない)
- 速報性を優先するため、**このファイルだけはPRレビュー不要・直接コミットでよい**(履歴はGitに残る)。※将来 main にブランチ保護をかける場合は、このファイルの扱い(保護の除外 or PR運用への変更)を決め直すこと
- ただし Claude が勝手に書き換えることは禁止。**追加・削除とも、Claude は公開情報(Microsoft公式のX・ステータスページ、SNSの障害報告)で傍証を集めた上でエントリ案を提示し、担当者の確認を得てからコミットする**。インシデントIDの確定は M365 管理センターのサービス正常性(担当者のみ閲覧可)で行う
- 編集前に必ず `git pull`。**同じ事象が既に登録されていないか確認してから**追加する(複数人が同時に登録しがち)
- 解消したらエントリを削除する(消した記録もGit履歴に残るので、後から「あの日の障害」を振り返れる)
- **発生から48時間を超えたエントリは「消し忘れ」の可能性を疑う。** inquiry-assistant は回答の根拠に使う前に継続中かを担当者に確認し、faq-reviewer は棚卸し時に1週間超のエントリを報告する
- 恒久的な知見が得られた場合(回避策が今後も使える等)は、削除時に faq-updater でFAQ化を検討する

## エントリのテンプレート

```markdown
## 🔴 対応中: <事象の一行説明>

- 発生: YYYY-MM-DD HH:MM / 情報源: (M365管理センターのインシデントID、社内連絡など)
- 影響: (誰の何ができない・遅いのか)
- 問い合わせへの回答方針: (個別調査の要否、利用者に伝える内容)
- 回避策: (あれば)
```
