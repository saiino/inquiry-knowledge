# FAQ Reviewer(FAQ定期棚卸し)

FAQの記述が Microsoft の現行仕様と食い違っていないかを定期的に点検するスキル。
**対象は `種別: Microsoft仕様` のエントリのみ。** `社内運用` は照合できる外部の一次情報が存在しないため対象外(AIが「変わったかも」と言い出してもノイズにしかならない)。
**どのケースでも Claude は merge しない。** 判断材料を揃えてPR/報告を出すまでが役割。

## 手順

### 1. 対象を選ぶ

0. まず `known-issues.md` を確認し、**発生から1週間を超えたエントリが残っていたら「消し忘れの可能性」として棚卸し報告に含める**(ファイル自体は書き換えず、担当者に継続中かの確認を促す)
1. `faq/index.md` を読み、件数が1以上のカテゴリファイルを読む
2. `種別: Microsoft仕様` のエントリを `最終確認日` が古い順に並べる
3. **1回の棚卸しで確認するのは5件まで**(レビュー可能な粒度を保つため)。ユーザーが件数やカテゴリを指定したらそれに従う

### 2. 1件ずつ突き合わせる

各エントリについて:

1. 「参考リンク」のURLを取得し、現行の記述を確認する
2. FAQの「原因」「解決方法」の記述と突き合わせる。確認するのは**事実の主張**(ライセンス条件、数値、手順、機能の有無)。文体や表現の違いは差分として扱わない
3. 日本語版Learnの記述が曖昧・古い可能性がある場合は、英語版(`en-us`)も確認する(日本語版は翻訳が遅れることがある)
4. 参考リンクがないエントリは、Microsoft Learn を検索して該当ドキュメントを探し、見つかれば参考リンクとして追加する

### 3. 結果ごとの処理

| 結果 | 処理 |
|---|---|
| ✅ 一致 | `最終確認日` を今日に更新(この行のみの変更なら他エントリ分とまとめて1つのPRでよい) |
| ⚠️ 不一致 | FAQの修正案を作成してPRに含める。**PR本文にLearn原文の引用とFAQ側の記述を並記**し、どちらの何が食い違うかを明示する |
| 🔗 リンク切れ/大幅改編 | FAQを書き換えず「要人間確認」として報告に載せる(移転先URLの候補があれば添える) |

**不一致時の鉄則:** 原文引用のない「変わったようです」は禁止。人間がAIの誤検知(表現の違いを仕様変更と勘違い等)と本物の仕様変更を区別できるように、必ず引用で示す。

### 4. PR と報告

1. ブランチを作り、コミットメッセージは `faq: 棚卸し YYYY-MM-DD(確認N件、修正M件)` 形式でPRを作成する。**mainへ直接pushしない**
2. PR本文に棚卸しサマリーを書く:
   - 確認したFAQ IDと結果(✅/⚠️/🔗)の一覧
   - ⚠️ の各件: Learn原文引用 vs FAQ記述、修正案の要点
   - 🔗 の各件: 状況と人間に確認してほしいこと
3. ユーザーには「⚠️と🔗の件数」を最初に伝える(全部✅なら「問題なし、最終確認日のみ更新」と一言で)

## 重要な注意

- FAQの技術的な正誤チェックが目的。誤字修正や文章の書き換えはしない(差分がノイズになりレビューしづらくなるため)
- 確認対象は Learn の現行記述のみ。ブログやフォーラムの「変わるらしい」という未確定情報でFAQを書き換えない
- 疑わしいが確証が持てない場合は ⚠️ ではなく 🔗(要人間確認)側に倒す
- `ステータス: サンプル` のエントリも棚卸し対象に含めてよい(書き方の見本として正確であるべきため)
