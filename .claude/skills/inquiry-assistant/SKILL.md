# Inquiry Assistant(問い合わせ回答支援)

保守担当者が受けた Power Platform の問い合わせに対して、FAQ を根拠にした回答案を作成するスキル。
**Claude はユーザー(問い合わせ元)に直接回答しない。** 必ず保守担当者がレビューして送信する前提の「回答案」を作る。

## 手順

### 1. FAQ を検索する

1. まず `faq/index.md` **だけ**を読む(FAQ全ファイルの一括読み込みは禁止。FAQが増えても検索コストを一定に保つため)
2. 問い合わせ内容からカテゴリを判断し、該当するカテゴリファイル(1〜2個まで)を読む
3. 症状・キーワード・タグが一致するFAQエントリを探す。表記ゆれ(英語/カタカナ、略称)は柔軟にマッチさせる

### 2-a. FAQ が見つかった場合

- そのFAQの「原因」「解決方法」を根拠に回答案を作成する
- 回答案には **FAQ ID を必ず明記**する(保守担当者が根拠を検証できるようにするため)
- FAQの内容と問い合わせの状況に差分がある場合(バージョン違い、前提の違いなど)は、その差分を「確認ポイント」として挙げる
- 最後に「**FAQどおりで解決したら報告は不要です。FAQと違う結末になったら教えてください**(FAQを修正します)」と一言添える

### 2-b. FAQ が見つからなかった場合

- Microsoft Learn を第一の情報源として Web 検索・調査する(learn.microsoft.com を優先。個人ブログ等は補助のみとし、使う場合は出典を明示)
- 調査結果をもとに回答案を作成し、**参照したURLを必ず併記**する
- 回答案の冒頭に「⚠️ FAQ未登録。Microsoft Learn 調査ベースの回答案」と明記する
- 最後に「この問い合わせがクローズしたら、原因と解決方法を教えてください。FAQ追加のPRを作ります」と伝える

### 3. 出力フォーマット

必ず以下の形式で出力する。

```markdown
## 回答案

(問い合わせ元にそのまま送れる文面。宛先に合わせた丁寧な日本語)

## 根拠

- FAQ: FAQ-PBI-APP-001(または「FAQ未登録」+ 参照URL)

## 確信度と確認ポイント

- 確信度: 高 / 中 / 低
- 送信前に確認すべき点(環境依存の前提、FAQとの差分、追加でヒアリングすべき事項)
```

## 追加質問が来た場合(表現調整)

回答後に利用者から「わかりにくい」「つまりどういうこと?」と追加質問が来たら:

1. より平易な表現・具体例(例え話や画面上の操作名)を使った回答案を作り直す
2. その表現で利用者が納得して解決したら、「FAQの記述も今の表現に更新しますか?」と提案する(追加質問が出た = FAQの説明がわかりにくかったシグナルなので、FAQ側にも還元する。更新は faq-updater スキルで行う)

## 重要な注意

- **推測で断定しない。** 情報が足りない場合は、回答案の中に「ヒアリング事項」として質問文を入れる
- 根拠にするFAQが `種別: Microsoft仕様` で、`最終確認日` から6か月以上経っている場合は、回答案を出す前に参考リンク先の現行記述を確認する。食い違いがあれば回答案は現行仕様に合わせ、確認ポイントに「FAQが古い可能性(棚卸し推奨)」と書く。特にライセンス・価格系は変更が多いので注意
- 確信度が「低」の場合はその理由を明記する(保守担当者が裏取りの要否を判断できるように)
- 社内固有の環境情報(テナント設定、容量の契約状況など)は Claude にはわからない。該当しそうな場合は確認ポイントに挙げる
- このスキルでは FAQ ファイルを**変更しない**(FAQ更新はクローズ後に faq-updater スキルで行う)
