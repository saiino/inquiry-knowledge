# Onboarding(新規参画者の受け入れ)

新しくチームに入った保守担当者が、説明会なしでこの仕組みを使い始められるようにする対話型オンボーディング。
所要時間の目安は30分。一度に全部やらず、ユーザーのペースに合わせて区切ってよい。

## 進め方

### ステップ1: 全体像(5分)

README.md の内容をベースに、次の3点だけをまず伝える(詳細は聞かれたら):

1. Claude は回答の「下書き係」。ユーザーへの回答はあなた(担当者)がレビューして送る
2. 話しかけ方は5パターンだけ(問い合わせを貼る / クローズ報告 / 障害報告 / 棚卸し / 月次サマリー)
3. FAQの変更は全部PRになるので、壊す心配はしなくていい

### ステップ2: ハンズオン(15分)

実際に体験してもらう。**練習であることを明示し、練習で作ったPRはマージせずクローズする。**

1. **回答案の練習**: サンプルFAQに載っている事例(例: 「アプリを共有したのに見られないと言われた」)を問い合わせとして貼ってもらい、inquiry-assistant の回答案(根拠・確信度つき)を体験してもらう
2. **クローズ報告の練習**: 架空のクローズ報告(例: 「原因は対象ユーザー未設定、追加と再発行で解決した」)をしてもらい、faq-updater がPRを作るところまで見せる。PRの中身(差分・本文)の読み方を説明し、**練習用なのでクローズする**
3. 報告の基準(FAQどおりで解決→報告不要)をここで体感として伝える

### ステップ3: 安全ルール(5分)

README の「保守担当向け: 安全に使うために」の3つの約束を確認してもらう:

1. `--dangerously-skip-permissions` を使わない
2. 削除系コマンドを「常に許可」に登録しない
3. `.claude/settings.json` と `CLAUDE.md` を勝手に緩めない(変更はPRで)

加えて、問い合わせメールには変な指示が混ざっていることがある(プロンプトインジェクション)ので、「回答案が問い合わせと関係ない動きをしたら、メール本文を疑って他の担当者に相談する」ことを伝える。

### ステップ4: 質問タイム

- 「なぜこうなっているの?」系の質問には **docs/design-principles.md を一次情報**として答える。書かれていない「なぜ」を聞かれたら、git log や PR の履歴から経緯を調べて答え、**答えられた内容は design-principles.md への追記PRを提案する**(次の参画者のため)
- **履歴から再構成した理由は必ず「推測」と明記し、断定しない。** 追記PRにも推測である旨を書く(履歴に残るのは「何を変えたか」で「なぜ」は薄い。AIがもっともらしく再構成した理由を、本当の設計意図と区別できないまま公式の思想として定着させないため)
- 操作の詳細を聞かれたら該当スキル(.claude/skills/)の内容を平易に説明する

## 重要な注意

- このスキルではFAQや設定を変更しない(ハンズオンで faq-updater が作る練習PRを除く)
- 相手はこの仕組みを初めて見る人。専門用語(PR、ブランチ等)には一言説明を添える。Gitに不慣れでも「レビューしてマージボタンを押す」だけで運用に参加できることを強調する
- オンボーディング完了時に「困ったらいつでも『これってどういう仕組み?』と聞けば答える」と伝える(人間の先輩の代わりに、いつでも聞ける相手がいることが定着の鍵)
