# FAQ Updater(FAQ更新PR作成)

クローズした問い合わせの対応結果を受け取り、FAQ更新のPRを作成するスキル。
**merge の判断は必ず人間(保守担当者のレビュー)が行う。** Claude は PR 作成まで。

## 対象となる報告

| 報告 | 処理 |
|---|---|
| FAQに無かった問い合わせ(未知)がクローズした | FAQ を新規追加 |
| 既存FAQで回答したが、実際は原因・解決方法が違った | 既存FAQ を修正 |
| 追加質問を受けて回答表現をわかりやすくした | 既存FAQ の表現を更新(内容は変えず伝わる書き方に) |

**「FAQどおりに解決した」場合は報告不要**(FAQに変化がないため)。報告が必要なのは「FAQに新しく入れる/直すことがあるとき」だけ、が運用ルール。

## 手順

### 1. 内容の確認

報告から以下が読み取れるか確認し、足りなければユーザーに聞く。

- 問い合わせの症状(利用者から見た事象)
- 最終的な原因
- 解決方法(手順として再現できる粒度)

**報告の形式は自由。** 対応時のメールやチャットのやり取りをそのまま貼ってもらってよい。その場合は Claude が上記3点を抽出し、足りない部分だけを質問する(担当者に整形の手間をかけさせない)。対応中の会話(inquiry-assistant で回答案を作った履歴)があればそれも活用してよい。

### 2. 追加か更新かを判断する

1. `faq/index.md` からカテゴリを特定し、該当ファイルを読む
2. **同じ原因のFAQが既にある場合は、新規追加せず既存エントリを更新する**(解決方法の追記、タグ追加、更新日の変更)。類似FAQの乱立はナレッジの検索性を下げる
3. 新規の場合は、そのカテゴリの既存IDの最大値+1で採番する(`faq/index.md` のID規則に従う)

### 3. FAQ エントリを書く

以下のテンプレートに従う。

```markdown
## FAQ-<ID>: <利用者の言葉に近い症状タイトル>

- 登録日: YYYY-MM-DD / 更新日: YYYY-MM-DD
- 種別: Microsoft仕様 / 最終確認日: YYYY-MM-DD
- タグ: `キーワード1` `キーワード2`

### 症状 / 問い合わせ内容

(利用者から見た事象。検索でヒットしやすいよう、利用者が使った表現を残す)

### 原因

(技術的な原因。1〜3文)

### 解決方法

(番号付き手順。次に同じ問い合わせが来たとき、別の担当者がそのまま実行できる粒度)

### 参考リンク

- (Microsoft Learn 等。あれば)
```

**書き方の注意:**

- `種別` は次の2択。**Microsoft仕様**(ライセンス、機能の挙動、画面操作など、Microsoft側の変更で古くなりうる知識。faq-reviewer スキルの定期棚卸し対象になる)/ **社内運用**(申請フロー、窓口、社内ルールなど。照合できる外部の一次情報がないため棚卸し対象外)。1エントリに両方混ざる場合はエントリを分ける
- `最終確認日` は、参考リンク等の一次情報と突き合わせて内容が正しいと確認した日。登録時は登録日と同じでよい
- タイトルは「利用者の困りごと」で書く(×「アプリの対象ユーザー設定について」 ○「アプリを共有したのに相手が閲覧できない」)
- 社内固有の情報(個人名、具体的なテナント名、メールアドレス等)は書かない。部署名・役割名程度に抽象化する。**メールの貼り付けから抽出する場合も同様に必ず除去する**(PRレビューでも人間が再確認する)
- 1エントリ1論点。原因が複数あった場合は「原因」内で場合分けするか、エントリを分ける

### 4. index.md を更新する

- 該当カテゴリの件数を +1 する(新規追加の場合)
- 新カテゴリが必要な場合は、ファイル作成+カテゴリ表・略号表への行追加も同じPRで行う

### 5. PR を作成する

1. `faq/` 配下の変更のみを含める(terms.json など他のファイルを混ぜない)
2. コミットメッセージ: `faq: <FAQ ID> <タイトル> を追加`(更新の場合は `を更新`)
3. ブランチに push し、PR を作成する。**mainへ直接pushしない**
4. PR本文には以下を書く:
   - 問い合わせの概要(1〜2行)
   - 追加/更新したFAQ ID
   - レビューしてほしい観点(手順の正確性、社内固有情報が漏れていないか)
5. ユーザーに PR の URL を伝え、「レビューして merge してください」と案内する

## 重要な注意

- 原因が確定していない(「たぶん◯◯」)状態でFAQ化しない。「原因は確定していますか?」と確認する
- ユーザーの報告と Microsoft の公式情報が食い違う場合は、PR本文にその旨を書いてレビュアーの判断を仰ぐ
- FAQの削除・大規模な再編成は、このスキルの範囲外。ユーザーと相談してから別途行う
