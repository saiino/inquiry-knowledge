# inquiry-knowledge — 問い合わせナレッジエージェント(PoC)

Claude Code を **問い合わせ回答ツールではなく、保守担当者(約5名)の支援ツール**として使い、
問い合わせ対応のたびに FAQ を自動で育てていく仕組みの PoC。

## 背景・課題

現在の Power Platform(Power BI など)の問い合わせ対応には次の課題がある。

- 過去に回答した内容を探すのに時間がかかる
- 同じような問い合わせが繰り返し発生する
- ナレッジが担当者に属人化しやすい
- FAQ を更新する運用が定着しにくい

AI のスキル(手順書)を介して対応を標準化すれば、担当交代があっても回答の一定性を保てる。

## 全体フロー

```text
問い合わせ
    │
    ▼
保守担当 ──(問い合わせ内容を貼り付け)──▶ Claude Code
    │                                       │
    │                     [inquiry-assistant スキル]
    │                                       │
    │                             faq/index.md を読む
    │                                       │
    │               ┌───────────┴───────────┐
    │            FAQ がある               FAQ がない
    │               │                       │
    │        FAQ を根拠に回答案      Microsoft Learn 等を
    │        (FAQ ID を明記)         調査して回答案(出典明記)
    │               └───────────┬───────────┘
    │                                       │
    ◀──────── 回答案 + 根拠 + 確信度 ────────┘
    │
 担当者が内容を確認・修正
    │
    ▼
 ユーザーへ回答 → 問い合わせ完了
    │
    ▼
 担当者が「最終的な原因」「解決方法」を Claude へ報告
    │
    ▼                        [faq-updater スキル]
 Claude が FAQ 更新 PR を作成(追加 or 既存更新を判断)
    │
    ▼
 保守担当がレビュー → Merge → FAQ が育つ
```

## リポジトリ構成

| パス | 役割 |
|---|---|
| `faq/index.md` | カテゴリ一覧とFAQ ID規則。Claude はまずここだけを読む |
| `faq/powerbi/` ほか | カテゴリ別のFAQ本体(Markdown) |
| `.claude/skills/inquiry-assistant/` | 問い合わせ → FAQ検索 → 回答案作成の手順書 |
| `.claude/skills/faq-updater/` | クローズ報告 → FAQ 追加/更新 → PR 作成の手順書 |
| `.claude/skills/faq-reviewer/` | FAQ定期棚卸し(Microsoft Learn現行記述との突き合わせ)の手順書 |

## 設計上のポイント

### 1. Claude は直接回答しない

```text
Claude → 保守担当 → ユーザー
```

必ず人がレビューしてから送るため、品質を担保できる。FAQ 更新も PR + レビュー必須。

### 2. index → 該当ファイルのみ読む

`faq/index.md` でカテゴリにルーティングし、必要なファイルだけを読む。
FAQ が増えても全件読み込みが発生せず、スケールする。

### 3. スキル = 標準化された手順書

対応手順・出力フォーマット・注意事項を `.claude/skills/` の SKILL.md に固定してあるため、
**誰が Claude に依頼しても同じ型の回答案・FAQ が出てくる**。担当交代時も運用が変わらない。

### 4. FAQ は Git 管理

- 変更履歴が全部残る(いつ・誰が・なぜ直したか)
- PR レビューで品質を担保
- Markdown なので人間もそのまま読める

## 使い方

### 問い合わせが来たとき

Claude Code に問い合わせ内容を貼り付ける。

> 問い合わせが来ました。「アプリを共有したのに、相手が権限エラーで見られないと言っています」

→ 回答案・根拠(FAQ ID または参照URL)・確信度が返ってくるので、確認してユーザーへ回答する。

### 問い合わせがクローズしたとき

**報告の基準: FAQどおりで解決したら報告不要。報告するのは「FAQに入れる/直すことがあるとき」だけ。**

| ケース | 報告 | Claude の動き |
|---|---|---|
| 既存FAQどおりで解決 | 不要 | — |
| FAQに無かった問い合わせ(未知)が解決 | する | FAQ 新規追加の PR |
| FAQで回答したが実際は原因が違った | する | 既存FAQ 修正の PR |
| 追加質問が来て、表現をわかりやすく直した | する(任意) | FAQ の表現を更新する PR |

報告の形式は自由。一言でもいいし、**対応時のメールのやり取りをそのまま貼ってもいい**(Claude が症状・原因・解決方法を抽出し、FAQ化の際に個人名などの社内固有情報は除去する)。

> さっきの問い合わせクローズしました。原因はアプリの対象ユーザー未設定で、対象ユーザー追加+アプリ再発行で解決。

→ FAQ 更新 PR が作られるので、レビューして merge する。

### FAQ の定期棚卸し(月1回目安)

Claude Code に依頼する。

> FAQ棚卸しして

→ `種別: Microsoft仕様` のFAQ(最終確認日が古い順に5件まで)を Microsoft Learn の現行記述と突き合わせ、一致なら最終確認日を更新、不一致なら**Learn原文の引用つき**で修正PRを作る。`種別: 社内運用` のFAQは照合先の一次情報がないため対象外(社内ルールの変更は人間がFAQを直す)。

ライセンス条件など Microsoft 側の仕様は変わるため、FAQ が「登録時は正しかったが今は違う」状態になるのを防ぐ仕組み。ここでも merge 判断は人間が行う。

## 将来的な構想(PoC の範囲外)

FAQ が十分育ったら、一般社員向けへの展開も考えられる。

```text
GitHub Enterprise → GitHub Actions → SharePoint → Copilot Studio(社内Bot)
```

ただし PoC では保守担当向け支援に限定する。

## 期待する効果

- 問い合わせ対応時間の短縮(過去回答の検索を Claude に任せる)
- FAQ の継続的な蓄積(クローズ報告 → PR がワンアクション)
- Git による履歴管理・レビュー
- 属人化の防止・担当交代時の回答一定性
- 将来的な Copilot Studio 等への展開余地
