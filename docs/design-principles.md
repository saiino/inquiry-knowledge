# 設計思想(なぜこうなっているか)

この仕組みの各ルールには理由がある。ルールだけ覚えて理由を知らないと、不便に感じた部分を緩めた結果、安全や品質の前提が崩れる。
**変更したくなったら、まずここで「なぜそうなっているか」を確認し、理由ごと否定できるならPRで変更を提案すること。**
新規参画者の「なぜ?」に Claude が答えるときの一次情報でもある。

## Q. なぜ Claude が問い合わせ元に直接回答しないのか

AIの回答は流暢だが、間違っていても流暢なので、受け取った側は気づけない。保守担当者が間に入って検証することで品質を担保する。「Claude → 保守担当 → ユーザー」の二段構えはこの仕組みの根幹で、崩すと品質保証がなくなる。

## Q. なぜ Copilot Studio や既製のKBツールではなく Git + Markdown なのか

1. **更新ループの摩擦を最小化するため。** ナレッジ管理が死ぬのは検索性ではなく「誰も書かない」から。クローズ報告一言でAIがFAQ整形からPRまでやる構成は、Git操作・Markdown編集・PR作成が得意な Claude Code だから成立する
2. **検証可能性のため。** FAQはPRレビューを通り、変更履歴が全部残り、出典リンクで一次情報に遡れる。Copilot Studio は渡されたナレッジを信じて配信するだけで、ソースの検証はしない
3. **可搬性のため。** Markdown はどこへでも持っていける。将来 SharePoint / Copilot Studio に配信する場合も、このリポジトリが正(ソース・オブ・トゥルース)のまま配信層を足せばよい

## Q. なぜ index.md から該当ファイルだけを読む方式なのか

FAQが増えても検索コストを一定に保つため。全件読み込みはFAQが数百件になった時点で破綻する。この方式なら数百〜1,000件程度までスケールし、それを超えたら埋め込み検索を「上に足す」(Markdownをやめる必要はない)。

## Q. なぜ index.md の件数列は「目安」なのか

複数のFAQ追加PRが同時に開くと件数列は必ずコンフリクトするため、「正はカテゴリファイルの中身、件数は目安」と割り切っている。衝突したら実エントリ数を数え直して解決する。

## Q. なぜ FAQ ID の欠番を詰めないのか

過去の回答メールに「FAQ-PBI-APP-002 参照」と書かれている可能性があるため。IDは一度発行したら永久欠番。

## Q. なぜ known-issues.md だけ PR 不要なのか

障害の速報がレビュー待ちで数時間止まると本末転倒だから。ただし代償として「レビューを通らない唯一の経路」になっているため、(1) 公開情報での傍証調査、(2) コミット前のユーザー確認、(3) 48時間超エントリの鮮度チェック、の3つのガードを義務にしている。このガードを外すなら PR 運用に戻すこと。

## Q. なぜ「FAQどおりで解決したら報告不要」なのか

報告義務を「FAQに入れる/直すことがあるとき」だけに絞り、運用の摩擦を最小にするため。ナレッジ運用の死因1位は報告のサボりであり、義務を減らすほど定着する。ただし「FAQと違う結末になった」は必ず報告する(FAQの間違いが直る唯一の経路のため)。

## Q. なぜ 種別(Microsoft仕様 / 社内運用)を分けるのか

AIによる定期棚卸しが成立するのは「照合できる一次情報(Microsoft Learn)」がある知識だけだから。社内運用には照合先がなく、AIに点検させると「変わったかも」という誤検知ノイズしか生まない。だから棚卸し対象を Microsoft仕様 に限定している。

## Q. なぜ棚卸しの不一致報告に「原文引用」を義務づけるのか

AIの誤検知(表現の違いを仕様変更と勘違い)と本物の仕様変更を、人間が区別できるようにするため。引用のない「変わったようです」は判断材料にならない。

## Q. なぜ .claude/settings.json に削除系コマンドの拒否設定があるのか

スキル(自然言語の指示)は確率的で、100%は守られない。設定によるブロックは機械的に発動する。「AIが読み違えても実行自体が止まる」層を1つ持つため。**この拒否リストを緩めるときは必ずPRでチーム合意を取る。**

## Q. なぜ「確信度: 高/中/低」ではなく「根拠の強さ: A〜D」なのか

AIの自己申告の確信度は較正されていない(間違っているときほど自信を持つことがある)ため、担当者が「高」を見て裏取りを省く——**一番危ないときに一番信用される**——という逆効果を生む。根拠の強さ(FAQ完全一致/部分一致/Learn調査のみ/根拠なし)は事実の分類なので、AIが偽れず人間が検証できる。

## Q. なぜ merge は必ず人間なのか

この仕組み全体の安全性・品質は「人間が最後に見る」に賭けている。AIのPRは体裁が整っているため、読まずに承認する誘惑が強い(自動化バイアス)が、それをやるとFAQの間違いが検証なしで蓄積し、将来の自分たちが間違った回答をする。FAQは「自分たちが次に使う道具」であり、レビューは未来の自分への投資。
